# 阿部接骨院 予約システム（Ver.1）

一般的な「予約時間＝担当者拘束時間」ではなく、**工程(Step)単位**で予約を管理する
Web 予約システムです。患者の来院時間・担当者の施術時間・機器の使用時間が一致しない
という当院の実態に合わせて設計しています。

将来的に既存の電子カルテ（`../emr`）へ連携できるよう、同じ技術構成
（Next.js + Supabase）で作成し、`patients` / `staff` テーブルを共有できる設計です。

---

## 最重要コンセプト：工程(Step)単位の予約

1件の予約（メニュー）は複数の**工程**で構成されます。各工程は次を持ちます。

- 工程名（例：全身通電／施術）
- 所要時間（分・5分単位）
- 順番
- **担当者を使用するか**（あり／なし）
- **使用機器**（なし／ハイチャージ 等）
- 同時利用人数（機器の占有数）

予約可否は「担当者の空き」と「機器の空き」を **別々に集計** し、組み合わせて判定します。

### 例：全身通電30分 → 施術30分（11:00 来院）

| 工程       | 時間          | 担当者 | 機器           |
| ---------- | ------------- | ------ | -------------- |
| ① 全身通電 | 11:00〜11:30  | なし   | ハイチャージ   |
| ② 施術     | 11:30〜12:00  | あり   | なし           |

→ 担当者予約表では **11:00 ○ / 11:30 × / 12:00 ○**。
患者には「11:00 予約」として表示します。

---

## 予約可否の判定条件

以下をすべて満たした場合のみ予約可能です（`check_booking_availability`）。

1. 担当者を使用する工程で、担当者が空いている
2. 使用機器が必要な工程で、機器に空きがある（同時利用人数 < 定員）
3. 勤務時間内（予約全体が担当者の勤務枠に収まる）
4. 休診ではない（院全体休診／担当者個別休、終日／時間帯）
5. 既存予約と重複しない

判定は **5分単位**、表示は **30分単位**（20分工程にも対応）。

---

## 二重予約防止

予約確定は Postgres の関数 `book_appointment` で行い、

- 同日について `pg_advisory_xact_lock` で処理を直列化
- 確定直前に `check_booking_availability` を **再判定**
- 予約本体（`appointments`）と工程実体（`appointment_steps`）を原子的に登録

することで、同時アクセス時も二重予約になりません。

## 主な機能

### 患者予約画面（`/`・公開・スマホ優先）

`① メニュー → ② 担当者・日時 → ③ お客様情報 → ④ 確認 → ⑤ 完了`

- 担当者ボタン（阿部・澁谷・萩原・林）は**画面遷移せず週間カレンダーだけ切替**
- 週間カレンダー：横＝日付／縦＝時間、○＝予約可・×＝空きなし・·＝受付時間外
- お客様情報は端末に保存し、2回目以降は自動入力（電話番号で名寄せ）

### 管理画面（`/admin`・要ログイン）

- 予約表：縦＝時間／横＝担当者、予約カード（患者名・メニュー・来院時間）
  - 担当者カラー（阿部:青／澁谷:緑／萩原:紫／林:オレンジ／勤務外:グレー）
  - ハイチャージ等の**機器の占有状況も併せて表示**
- 予約追加／変更／キャンセル、患者検索・新患登録
- メニュー・工程設定（`/admin/services`）：自由に作成・編集
- 休診設定（`/admin/closures`）

## データ構造

| テーブル             | 役割                                                     |
| -------------------- | -------------------------------------------------------- |
| `patients`           | 患者基本情報（emr と共有可）                             |
| `staff`              | 担当者（色・並び順・受付可否。emr と共有可）             |
| `services`           | メニュー                                                 |
| `service_steps`      | メニューを構成する工程テンプレート                       |
| `equipment`          | 機器（ハイチャージ＝同時4名。追加可能）                  |
| `appointments`       | 予約（来院時刻・担当者・メニュー）                       |
| `appointment_steps`  | 工程の実体（開始/終了・担当者使用・使用機器・利用人数）  |
| `staff_schedules`    | 勤務時間（曜日ごと）                                     |
| `closures`           | 休診（全体／個別・終日／時間帯）                         |

`appointment_steps` が予約可否判定の中核データです。担当者を使う工程のみ `staff_id`、
機器を使う工程のみ `equipment_id` を持つため、担当者占有と機器占有を別々に集計できます。

## 技術構成

- Next.js 14（App Router）/ React 18 / TypeScript / Tailwind CSS
- Supabase（PostgreSQL + Auth + RLS）
- 予約可否ロジックは 2 箇所に実装
  - `src/lib/booking.ts`：カレンダー描画用（クライアント側の目安）
  - `supabase/schema.sql` の関数：確定時の正となる再判定（原子的）

## セットアップ

### 1. Supabase

emr と同じプロジェクトを使うと将来のカルテ連携が容易です（別プロジェクトでも可）。

1. Supabase の SQL Editor で `supabase/schema.sql` を実行
2. 続けて `supabase/seed.sql` を実行（担当者4名・メニュー・機器・勤務時間・デモ予約）
3. 管理画面ログイン用に Authentication → Users でスタッフを1名作成
   - emr と共有している場合は emr のログインをそのまま使えます

### 2. ローカル起動

```bash
cd booking
cp .env.example .env.local   # URL と anon key を記入
npm install
npm run dev                  # http://localhost:3000
```

- 患者予約：`/`
- 管理画面：`/admin`（未ログインは `/admin/login` へ）

### 3. 本番公開

Vercel に `booking` ディレクトリを Root Directory として指定し、環境変数
`NEXT_PUBLIC_SUPABASE_URL` / `NEXT_PUBLIC_SUPABASE_ANON_KEY` を設定して deploy します。
